要約 (TL;DR)

4 候補（ClickHouse / BigQuery / Snowflake / DuckDB）のうち、当社のクエリ支配的なワークロードと月額予算（〜¥45万）では ClickHouse を推奨。既存 Postgres を OLTP として残し、CDC で ClickHouse に取り込む併用構成とする。

  • 集計クエリの P95 は現行比 11〜18 倍 改善（§5 検証ログ）。
  • 月額試算は最安だが、運用負荷とマネージド度は中程度（要 SRE 体制）。
  • フルマネージドを最優先する場合は BigQuery が次点。

背景と課題

現行基盤の状況

現行の分析基盤は OLTP 用の Postgres を相乗りで利用しており、データ量の増加に伴い分析クエリと業務トランザクションが競合している。特に オンプレ Postgres では月次バッチが SLA を逸脱し始めている1状況で、四半期の経営レポート生成が平均 42 分かかっている[3]。

調査の前提と制約

本調査では、列指向の分析特化エンジンへ分析ワークロードを切り出すことを前提に、初期コスト・運用負荷・既存資産との接続性を主要な制約として候補を評価する。

対象データとワークロード

  • 対象データ量: 常時 ~4 TB、月間スキャン ~12 TB。
  • 主要ワークロード: ダッシュボード集計（高頻度・低レイテンシ要求）。

既存環境の制約

  • 制約: 既存 BI ツール（Metabase / Looker）との互換、SQL 資産の流用。

評価方法と軸

テストデータセット

各候補に対し、代表的な集計クエリ 14 本・取込ジョブ 3 本を同一データセット（TPC-DS 100GB 相当 + 自社サンプル）で実行し、レイテンシ・スループット・月額コストを実測した。定性軸は 5 段階で評価している。

評価軸の重み

意思決定では クエリ性能（集計）と月額コストを最重視し、運用負荷を次点に置いた。マネージド度は「SRE が 1 名で運用可能か」を基準に判定している。

定量指標の算出方法

レイテンシは P95、スループットは 60 秒間の完了クエリ数で測定し、各 5 回試行の中央値を採用した。月額コストは後述（§4）の前提に基づく税抜試算。

定性指標の評点基準

運用負荷・ロックイン・学習コストは、社内の運用実績とドキュメント整備度をもとに 5 段階で相対評価している。

コスト試算の前提

ワークロード想定

月額はいずれも スキャン 12 TB/月・ストレージ 4 TB 常時・同時実行 8 を前提に試算した。クエリ量が支配的なワークロードのため、スキャン従量・クレジット定額型は相対的に割高になる。

!
注意 — 自前運用の隠れコスト

ClickHouse の月額には SRE 工数（試算 0.3 人月）を含めていない。マネージド ClickHouse Cloud を選ぶ場合、月額は約 ¥420k に上昇し BigQuery と拮抗する。

i
補足 — 為替前提

USD 建て課金は ¥158/USD で換算。為替変動 ±10% で BigQuery / Snowflake の順位は変わらない。

推奨と結論

当社のクエリ支配的ワークロードと予算制約のもとでは、推奨は ClickHouse + 既存 Postgres の併用3とする。CDC 同期遅延は P95 8 秒で SLA 内に収まることを確認済み（§5）。

✓ 推奨

ClickHouse（自前運用）+ Postgres CDC 併用。初期は IaaS 上の 3 ノード構成で開始し、SRE 体制が整い次第マネージドへの移行を再評価する。フルマネージド最優先なら BigQuery を次点とする。

レビューブロックの状態別デザイン

未対応 open

本文ハイライト=塗り + 2px 下線（青）。カードは左 3px 青、不透明度 100%。返信入力＋解決ボタンを表示。

返信あり reply

ハイライト=塗り + 2px 下線（琥珀）。カードに返信スレッド表示（agent 返信はグラデ avatar）。未解決として残る。

解決済み resolved

ハイライトは後退（破線下線のみ・塗りなし）。カードは不透明度 78% に落とし、解決者バナー表示。フィルタで非表示可能。

状態差は 色相 + 塗りの有無 + 不透明度で表現し、文言の追加に頼らない。未対応＝主張、返信あり＝進行中、解決済み＝後退、という視覚的ヒエラルキー。

ブロック方針
