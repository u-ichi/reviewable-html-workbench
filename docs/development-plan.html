Reviewable HTML Workbench 開発計画

まずCodex CLIでHTMLレビュー用pluginを完走させ、その後にClaude Code対応を追加するための計画。

目的

reviewable-html-workbench を、まずCodex CLIで実用できるpluginとして完走させる。Claude Code対応はCodex版MVPが実出力評価まで通った後に追加する。

最初の到達点は、Codex上で設計資料をレビュー可能なHTMLとして生成し、preview URLを提示し、HTML上のコメントを読み込み、fixture上で分類・agent返信・処理状態保存を検証できる状態にすること。自由なdocument model反映やUI検証の強化は、最小縦通し後のhardeningで扱う。

開発方針

  • SKILL.md は薄く保ち、判断順・ガード・呼び出し手順だけを書く。
  • 重い処理は scripts/html_review_workbench/ のCLIとして実装する。
  • 開発はCodex firstで進める。Claude対応は最後の互換化フェーズで追加する。
  • Claude / Codex 差分は plugin manifest と最小adapterに閉じ込める。
  • HTML comment は comments.json を正本にし、localStorage はstandalone用の補助に留める。
  • Preview Runtime は Tailscale IPv4 を優先し、fallback は 127.0.0.1。0.0.0.0 は禁止する。

優先順位

  1. Milestone 0: CIとmanifest整理
  2. Milestone 1: schema固定
  3. Milestone 2: HTML Renderer MVP
  4. Milestone 4: Preview Runtime
  5. Milestone 5: Comment UI and Store
  6. Milestone 6: Review Ingestion最小縦通し
  7. Milestone 6.5: Review Ingestion Hardening（backlog #12）
  8. Milestone 7-9: Codex skill統合、Codex packaging、実出力評価
  9. Milestone 9.5: Post Evaluation Hardening（backlog #13）
  10. Milestone 10: Claude Code対応
Milestone 3 の Diagram Planner は重要だが、最初のレビューサイクル成立には必須ではないため、Renderer MVP後に最小実装から進める。

backlog運用

実装はMilestone単位の docs/backlog/NN-*.md を起点に進める。各backlogには、作業中に開発計画と設計を同期するルールを明記する。

  • 作業開始時に対象backlog、docs/development-plan.html、docs/design.html の関連箇所を確認する。
  • 大きい進捗、Milestone順序、完了条件、リスクが変わった場合は docs/development-plan.html を同じ作業内で更新する。
  • skill範囲、script API、データ構造、Preview Runtime、コメント取り込み仕様が変わった場合は docs/design.html を同じ作業内で更新する。
  • 途中で設計が変わった場合、実装だけを進めず、開発計画と設計HTMLの両方を直してから進捗記録する。
  • /backlog progress では、更新した計画/設計セクション、検証結果、残った未決事項を記録する。
  • /backlog done は、対象backlogのゴール条件に加えて、開発計画と設計の更新要否を確認してから完了にする。

Milestone 0: 基盤整理

Done repoをplugin開発できる最低限の状態にする。

作業

  • .codex-plugin/plugin.json を優先して整える。.claude-plugin/plugin.json は存在だけ維持し、本格対応は最後に回す。
  • skills/*/SKILL.md のdescriptionを実使用トリガーに合わせて調整する。
  • scripts/html_review_workbench.cli のsubcommand公開仕様を固定する。
  • docs/design.html と docs/development-flow.md を現行仕様に同期する。
  • GitHub Actionsで unittest、Codex manifest JSON syntax check、plugin構成check を実行する。

固定した公開仕様

  • .github/workflows/test.yml は main push と pull request で unittest、plugin manifest JSON syntax check、CLI help check を実行する。
  • .codex-plugin/plugin.json は interface の表示名、説明、capabilities、defaultPrompt、brandColor を必須fieldとしてテストで確認する。
  • scripts.html_review_workbench.cli は check-model、render、preview、ingest-review、validate の5 subcommandを公開仕様として持つ。
  • skills/*/SKILL.md は Codex が発火判断に使えるように、Triggers と使用しない場面をdescriptionに含める。

完了条件

  • python3 -m unittest discover -s tests が通る。
  • python3 -m json.tool .codex-plugin/plugin.json >/dev/null が通る。
  • python3 -m scripts.html_review_workbench.cli --help が通る。

Milestone 1: Data Contract

Goal HTML生成・コメント・preview sessionの入出力schemaを固定する。

作業

  • schemas/document-model.schema.json で文書モデルの必須field、review settings、block構造を定義する。
  • schemas/comments.schema.json でコメントthread、status、agent reply構造を定義する。
  • schemas/preview-session.schema.json でpreview session manifestを定義する。
  • tests/fixtures/ に文書モデル、コメント、preview sessionの最小fixtureを置く。
  • scripts/html_review_workbench/schema_validation.py に標準ライブラリだけで動くschema validation helperを置く。

完了条件

  • tests/test_schema_validation.py で3 schemaとfixtureの対応を検証できる。
  • fixtureを使って後続Milestoneのrenderer、comment store、preview runtimeテストを始められる。

Milestone 2: HTML Renderer MVP

Goal 最小文書モデルから index.html と renderer-manifest.json を生成できる。

作業

完了条件

  • validate でHTML存在、manifest存在、asset存在、review block存在を検証できる。

Milestone 3: Diagram Planner

Milestone 4: Preview Runtime

Goal HTML生成後に、ユーザーがクリックできるpreview URLを提示できる。

作業

  • preview_server.py を実装する。
  • 空きport選択を実装する。
  • HTML_REVIEW_WORKBENCH_TAILSCALE_IP がある場合はそのIPv4を優先する。
  • tailscale ip -4 が成功した場合はTailscale IPv4にbindする。
  • Tailscale IPv4が取れない場合は 127.0.0.1 にbindする。
  • 0.0.0.0 bindを拒否する。
  • 成果物の annotations/preview-session-*.json にsession manifestを書く。
  • preview server が起動元 agent の親PIDを監視し、親プロセス終了時に自動終了するようにする。

完了条件

  • preview --mode auto がURL、PID、owner PIDを返す。
  • sandbox外で取得したTailscale IPv4を環境変数で渡せる。
  • Tailscaleなしのfixture testで 127.0.0.1 fallbackを確認できる。
  • 0.0.0.0 が入力されても拒否される。
  • 成果物の annotations/preview-session-*.json に session manifest が作成される。
  • 起動元親PIDが終了するとpreview serverが自動終了する。

Milestone 5: Comment UI and Store

Goal HTML上で範囲選択コメントを作成し、comments.json に保存できる。

作業

  • templates/review-comments.js を実装する。
  • Selection APIで選択範囲、block id、前後文脈を取得する。
  • comment_store.py を実装する。
  • comments.json の読み書きとschema validationを実装する。
  • standalone modeではlocalStorage export/importを提供する。
  • review-server modeでは comments.json を正本にする。
  • ヘッダーを含む生成HTML全域を data-review-block 化してコメント可能にする。
  • 範囲選択時は選択位置近くにNotion風のコメントUIを表示する。
  • 保存済みコメントの選択範囲を本文側でハイライトする。見出しから本文に跨る複数text node選択は anchor.start/end で復元する。
  • thread popoverはrole/kind表記を出さず、本文クリックで編集できる。初回コメントは Cmd+Enter / Ctrl+Enter またはCommentボタンで投稿し、返信欄は Enter で投稿、Shift+Enter で改行できる。
  • statusは needs_agent_review、needs_user_reply、resolved に絞る。needs_user_reply は青系、resolved は薄いグレー系、needs_agent_review は通常コメント色で表示する。
  • 06完了直前、07着手前にPreview URLでユーザーがコメント作成・保存・再読込を確認する。

Milestone 6: Review Ingestion

Goal 「レビュー終わったので確認して」発話からコメントを読み込み、分類・agent reply書き戻し・処理状態保存をfixtureで検証できる。

作業

  • ingest_review.py を実装する。
  • コメントを actionable, needs_clarification, blocked, already_addressed に分類する。
  • actionable はdocument modelへ直接反映せず、反映候補として annotations/review-cycle-state.json に記録する。
  • needs_clarification は同じコメントスレッドへagent replyを追加する。
  • annotations/review-cycle-state.json に処理結果を残す。
  • fixtureで comments.json → 分類 → agent reply追記 → state保存まで検証する。

完了条件

  • ingest-review CLIが comments.json を読み込み、分類結果を返す。
  • 確認が必要なコメントにHTML上のagent replyが残る。
  • 処理結果が review-cycle-state.json に残る。
  • document model自由反映、UI E2E、review-comments.js 分割、comments status拡張はこのmilestoneに含めない。

Milestone 6.5: Review Ingestion Hardening

Follow-up backlog #12として、07の最小縦通し後にUI検証、JS分割、状態設計、document model反映を安全に強化する。

作業

  • review-comments.js の最低限のDOM単位検証方針を決める。新規依存なしでは、まず主要責務境界の静的検証で固定する。
  • 物理分割はまだ行わず、comments-api、selection-anchor、thread-popover 相当の境界をテストで追跡する。次に肥大化した時に分割する。
  • comments UI statusとingestion classificationを混ぜない状態設計を維持する。
  • document model反映は、--apply-model 指定時のfixture上の完全一致置換から始め、自由文書への直接反映に拡大しない。
  • Milestone 9の実出力評価へ渡す確認観点を整理する。

完了条件

  • 自動検証に寄せる項目とユーザー確認に残す項目が分離されている。
  • document model反映の安全な適用範囲が設計に反映され、fixtureで検証されている。
  • 実出力評価で確認すべきレビューサイクル項目が整理されている。

Milestone 7: Codex Skill Integration

Goal Codex上で2つのskillが共通scriptを使って一連のworkflowを実行できる。

  • visual-html-renderer から check-model, render, validate, preview を呼ぶ手順を確定する。
  • reviewable-design-doc から check-model, render, ingest-review, preview を呼ぶ手順を確定する。
  • Codex pluginとしてのskill discovery、trigger、CLI呼び出し、成果物配置を確認する。
  • agents/openai.yaml をSKILL.mdと同期する。

Codex統合で固定する順序

  • visual-html-renderer は、raw textやMarkdown変換結果ではなく、HTML表現を前提に直接設計した文書モデルを check-model → render → validate → preview の順に共通CLIへ渡す。
  • reviewable-design-doc は同じ check-model → render → validate → preview に加え、レビュー完了後に ingest-review を呼ぶ。
  • agents/openai.yaml は trigger例、default prompt、共通CLI entrypoint、workflow順をSKILL.mdと同期する。
  • tests/test_codex_skill_integration.py で同じfixtureから同じHTML bundle構造が出ること、reviewable workflowだけが annotations/review-cycle-state.json を追加することを検証する。

Milestone 8: Codex Plugin Packaging

Goal Codex CLIでpluginとして読み込める。あわせて、既存のbase repo管理方式へどう組み込むかを決める。

  • Codex: .codex-plugin/plugin.json と skills/ の構成を検証する。
  • Codex marketplace登録方法を docs/codex-plugin-packaging.md にdocs化する。
  • Codex側でpluginを有効化し、2 skill が実際に発火することを確認する。
  • base repoの home/skills / home/generated/codex-skills 管理とplugin管理のどちらを管理元にするか決める。
  • plugin versioningとrelease tag手順を docs/codex-plugin-packaging.md に固定する。

base repo 管理方式との統合検討

現在のbase repoは、共通skillを home/skills/<name> に置き、Codexへは home/generated/codex-skills/ 経由で ~/.agents/skills/ にcopy配布する。plugin化する場合、この既存経路と衝突しない導入方式をCodex側で先に決める。Claude側の ~/.claude/skills / plugin install 方針はMilestone 10で扱う。

方式 内容 採用判断
base repo 管理へ取り込み home/skills/visual-html-renderer と home/skills/reviewable-design-doc を管理元にし、scriptはbase repo配下に置く。 Codex全プロジェクトへ即時展開したい場合に向く。plugin repoとしての独立性は下がる。
外部plugin repoとして管理 このrepoを管理元にし、Codexはmarketplace entryで参照する。Claude連携は最後に追加する。 採用。scriptが厚く、versioningとテストをrepo単位で持てる。base repo側には同名skillを置かない。
hybrid adapter base repo側には薄いadapter skillだけ置き、実処理と管理元skillはこのplugin repoを参照する。 移行期に向く。二重定義を避けるためadapterの責務を明文化する必要がある。

統合時の確認項目

  • plugin skill は marketplace cache のcopy配置で扱い、home/skills、home/generated/codex-skills、live ~/.agents/skills / ~/.codex/skills に同名skillを置かない。
  • bin/build-codex-skills.sh はbase repoの home/skills だけを入力にするため、このplugin repoをその配下へコピーしない限り生成物とは衝突しない。
  • base repoの install.sh にはCodex marketplace登録やplugin installを含めず、個別導入手順に分ける。
  • 既存skillからこのpluginを呼ぶ場合は、直接script呼び出しではなくplugin skill境界経由にする。

採用する登録方式

このrepoは plugin root と repo-local marketplace root を兼ねる。.agents/plugins/marketplace.json は reviewable-html-workbench-local marketplace を定義し、plugin entry は source.path: "./plugins/reviewable-html-workbench" を参照する。plugins/reviewable-html-workbench はrepo rootへのsymlinkで、Codex marketplace loaderの既存パターンに合わせるために置く。導入は codex plugin marketplace add /path/to/reviewable-html-workbench で行う。

marketplace policy は INSTALLED_BY_DEFAULT とする。AVAILABLE のままだと marketplace 登録後もplugin cacheが作られず、skill discovery に載らない。

repo path に空白が含まれる環境では、Codex CLIのsource判定がgit source側へ倒れる場合がある。その場合は空白を含まないsymlinkを作り、codex plugin marketplace add /private/tmp/reviewable-html-workbench のようにsymlink pathを渡す。

base repoの install.sh には含めない。Codex全プロジェクトに強制配布すると同名skillの二重登録リスクが高いため、まずは個別導入とし、必要になった場合だけbase repo側へ薄いadapterを追加する。

Milestone 9: Codex 実出力評価

Goal Codex上で実際の設計資料をこのpluginで作り、レビューサイクルが成立することを確認する。

  • docs/design.html 自体をpluginで再生成する。
  • 生成されたpreview URLを開いて表示確認する。
  • HTML上でコメントを入れる。
  • ingest-review でコメントを取り込む。
  • 確認が必要なコメントにagent replyが残ることを確認する。
  • 反映後HTMLを再生成する。
  • UI E2E、review-comments.js 物理分割、document model反映拡張の要否を評価receiptに記録し、Milestone 9.5へ引き継ぐ。

Milestone 9.5: Post Evaluation Hardening

Gate backlog #13として、実出力評価で見つかった不足をClaude対応前に必ず回収または明示的に見送る。

作業

  • 実出力評価receiptから、UI E2E、JS物理分割、document model反映、skill導線の不足を分類する。
  • Playwright等のブラウザE2Eを導入するか、導入しない場合の代替検証を決める。
  • review-comments.js を comments-api、selection-anchor、thread-popover などへ物理分割するか、維持基準を決める。
  • document model反映を完全一致置換から拡張するか、拡張しない場合の安全基準を決める。
  • 実施したhardeningと見送った判断を docs/design.html と docs/development-plan.html に反映する。

完了条件

  • 10の実出力評価で見つかった不足が未分類のまま残っていない。
  • UI E2E、JS物理分割、document model反映拡張が、実施済みまたは見送り理由付きで記録されている。
  • Claude対応へ進んでもよい残リスクだけが明示されている。

Milestone 10: Claude Code対応

Later Codex版MVPの実出力評価とPost Evaluation Hardeningが終わった後、Claude Code pluginとして同じworkflowを使えるようにする。

  • Claude plugin schemaに合わせて .claude-plugin/plugin.json を調整する。
  • claude plugins validate . を通す。
  • claude --plugin-dir <repo> で手元起動する。
  • Claude側のskill discovery、slash command、CLI呼び出し差分を確認する。
  • base repoの ~/.claude/skills copy配布と plugin install のどちらを管理元にするか決める。
  • Codexで確立したfixtureと実出力評価をClaudeでも再実行する。

リスクと対策

リスク 対策
Claude対応を早く入れてCodex MVPが遅れる Milestone 10までClaude対応を後回しにし、Codexでレビューサイクルを先に完走する。
Claude plugin schemaとCodex plugin schemaがずれる manifestを分け、共通項目だけを同期する。Claude差分はMilestone 10で吸収する。
base repo管理の home/skills とplugin repoのskillが二重に有効化される Milestone 8で正本を決める。推奨はplugin repoを正本にし、base repo側は必要なら薄いadapterだけにする。
SKILL.md が肥大化する 詳細はdocsとscriptへ逃がし、SKILL.mdは手順とガードに限定する。
localStorageコメントをagentが読めない review-server modeでは comments.json を正本にする。
preview serverが意図せず公開される 0.0.0.0 を拒否し、Tailscale IPv4または 127.0.0.1 のみ許可する。
コメント反映が勝手な設計判断になる Milestone 6ではdocument modelへ直接反映せず、actionable をstateへ記録する。自由反映はMilestone 6.5以降で限定置換から段階的に扱う。
comments UI statusとingestion分類が混ざる UI statusは needs_agent_review / needs_user_reply / resolved に留め、actionable などの分類は review-cycle-state.json 側で扱う。
コメントUIの手動確認依存が残る Milestone 6.5では静的境界検証までに留め、Milestone 9.5でPlaywright等のE2E導入または代替検証を必ず判断する。
実出力評価で見つかった不足が回収されない Milestone 9の完了条件に不足分類を入れ、Milestone 9.5 / backlog #13 で回収または見送り理由の記録を必須にする。
diagram生成が外部送信を伴う 外部図式レンダラーや画像生成は承認ゲート対象にする。

当面の次タスク

  1. Milestone 6で comments.json → 分類 → agent reply/state保存の最小縦通しをfixtureで検証する。
  2. Milestone 6.5でUI検証、JS分割、状態設計、document model反映のhardeningを行う。
  3. その後、Milestone 7-9でCodex skill統合、plugin packaging、実出力評価へ進む。
  4. Milestone 9.5で実出力評価の不足を回収してから、Milestone 10のClaude Code対応へ進む。
