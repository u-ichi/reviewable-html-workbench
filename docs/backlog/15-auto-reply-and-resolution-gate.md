# 15: コメント自動回答 + 解決待ちゲート

## 概要

ユーザーが HTML 上にコメントを追加したら、agent が自動的に検知して add-reply で回答する。コメントスレッドが解決するまでドキュメント修正には入らない制御をセットにする。

## 動機

現状はユーザーが「コメント入れた」と agent に伝え、agent が ingest-review → add-reply を手動で実行する。コメント追加から回答までのラグと手間を削減したい。

## 要件

### 1. コメント追加の自動検知 → agent 自動回答

- preview server が comments.json の PUT（新規コメント追加）を受けた時に、agent に通知する仕組みを追加する
- agent は通知を受けて ingest-review → add-reply を自動実行する
- 回答内容はコメントの `comment` と `selected_text` を読み、設計資料の文脈を踏まえた実質的な回答にする

### 2. 解決待ちゲート

- `needs_clarification` のコメントが未解決の間は、設計反映（ドキュメント修正）に進まない
- ユーザーが「解決」ボタンを押した時点で、そのコメントの指摘内容にドキュメント修正が必要ならば agent が自動的に反映する
- 解決 = 議論が合意に達した、という意味。修正不要な質疑応答だけで解決する場合もある
- SKILL.md のフロー制御として記述する

### 3. 修正完了のブラウザ通知

- agent がドキュメント修正 + 再 render を完了したら、ブラウザ上に通知を表示する
- ユーザーは通知を見て、自分のタイミングでリロードして再レビューに入る
- 自動リロードはしない（レビュー中の読み位置やコメント入力中の状態を壊さないため）
- 通知方式の候補: preview server から SSE (Server-Sent Events) で push / ポーリングで comments.json の更新タイムスタンプを監視
- 通知 UI: トースト or バナー（「ドキュメントが更新されました。リロードして最新版を確認できます」）

## 検討事項

- 通知方式: ファイル監視 / preview server のイベント出力 / ポーリング
- agent がループし続ける必要があるか、イベント駆動にできるか
- 複数コメントが同時に入った場合の処理順序
- agent セッションが切れた場合のリカバリ
