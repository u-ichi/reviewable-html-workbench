HTML出力系 skill 分割設計書

HTML出力を制御する共通レンダラー skill と、レビュー可能な設計資料を作る用途別 skill に分ける設計。既存HTMLの後処理ではなく、HTMLを出す時点の標準レンダリング経路を置き換え、生成後はセッション限定のプレビューURLまで提示する。

結論

設計は2つに分ける。visual-html-renderer は、既存のHTML出力制御を置き換える共通レンダラーとして、図・画像・レイアウト・HTML品質・プレビューURL提示を担当する。reviewable-design-doc は、そのレンダラーを使って、レビュー可能な設計資料を作る用途別 skill とする。

重要な境界は、レンダラーが内容判断を作らないこと。図示や画像は入力された主張・根拠・構造を理解しやすくするために使い、設計資料としての論点整理やレビュー導線は reviewable-design-doc 側が担当する。ブラウザを自動で開く責務は持たず、サーバー起動とURL提示までを担当する。

2つの skill への分割

visual-html-renderer

既存のHTML出力を制御している機能を置き換える共通レンダラー。HTMLを最終成果物にする skill は、個別にHTMLテンプレートを書くのではなく、このレンダラーへ文書モデルを渡す。

共通基盤図示判断HTML品質Preview URL

reviewable-design-doc

レビュー可能な設計資料を作る用途別 skill。要求、アーキテクチャ、意思決定、未決事項、レビュー観点を整理し、コメント可能なHTMLとして出力する。

用途別設計資料レビュー導線

分割理由

HTML出力全般の品質改善と、設計資料固有のレビュー体験を混ぜると責務が広がりすぎる。共通レンダラーと用途別 authoring skill に分けることで、既存のHTML出力系 skill にも横展開できる。

呼び出し関係

reviewable-design-doc は本文構成と設計レビュー観点を作り、最終HTML生成は visual-html-renderer に委譲する。他の調査・分析系 skill も同じレンダラーを使える。

完了条件

HTML出力系 skill が共通レンダラー経由で図示つきHTMLを出せること。設計資料 skill はコメント可能な設計HTMLを生成し、レビュー結果をJSONで再利用できること。生成後にクリック可能なプレビューURLが提示されること。

スコープ定義

2層アーキテクチャ

用途別 skill が文書モデルを作り、共通レンダラーがHTML表現を担当する。既存のHTML出力系 skill は、最終出力を直接書かずに visual-html-renderer を呼び出す形へ寄せる。

生成フロー

visual-html-renderer のフロー

  1. 文書モデルとレンダリングオプションを受け取る。
  2. 画像・図で説明可能なブロックを抽出する。
  3. Mermaid / SVG / draw.io / 画像生成の形式を選ぶ。
  4. 本文、図、表、callout、CSS、必要な review UI をHTMLへ組み込む。
  5. HTML構造、図の非空、モバイル表示、コメントUI部品を検証する。
  6. preview が有効なら空きportを選び、tailscale ip -4 が返る場合はそのIP、なければ 127.0.0.1 にbindする。
  7. セッション限定サーバーを起動し、URLと停止方法を提示する。ブラウザは自動で開かない。

reviewable-design-doc のフロー

  1. 設計対象、目的、読者、レビュー観点を整理する。
  2. 要求、制約、アーキテクチャ、意思決定、代替案、未決事項へ分ける。
  3. 図示したい論点、レビューコメント必須範囲、preview 要否を指定する。
  4. visual-html-renderer に文書モデルを渡す。
  5. コメントJSONを次の修正・合意形成の入力として扱う。確認が必要な場合は、チャットだけで聞かずHTML上の同じコメントスレッドへ返信を書き戻す。

図示判断ポリシー

図にする対象 推奨形式 理由 生成時の注意
処理順、判断順、承認フロー Mermaid flowchart + rendered SVG sourceが読みやすく、差分管理しやすい。 本文と矛盾しないよう、ノードは本文の見出しやfield名から生成する。
責務分離、システム構成、データ流 inline SVG または Mermaid HTML内で安定して表示でき、外部CDNに依存しない。 複雑すぎる場合は本文側へ分解し、1図1論点にする。
業務フロー、編集前提の図 .drawio + exported SVG 人間がdraw.ioで編集できる。 HTMLはSVGを表示し、draw.ioは編集用assetとして同梱する。
概念イメージ、比喩的な説明 生成画像 読者の直感理解を助ける。 数値、根拠、関係性の正確さが必要な図には使わない。
図示は「本文の構造を見える化する」用途に限定する。入力に存在しない因果関係、判断、優先順位を図で追加してはいけない。

範囲選択コメント設計

コメント機能は visual-html-renderer が提供するHTML部品として実装し、reviewable-design-doc が必須オプションとして有効化する。静的ファイル単体で動く standalone mode を標準にし、必要に応じてローカル保存用の review-server mode を追加する。agent が確認したい点は、チャットではなくHTML上の該当コメントスレッドへ返信する。

standalone mode

file:// で開ける。コメントは localStorage に保存し、JSON export/import で agent に渡す。

標準外部依存なし保存はブラウザ単位

review-server mode

Preview Runtime と同じbind方針の小さなサーバーで annotations/comments.json に保存する。Tailscale IP が使える場合はVPN越しのSSH/tmux環境からも同じURLで開ける。

任意永続JSONVPN限定可

コメント保存データ

  "document_id": "html-output-skill-split-design",
  "block_id": "diagram-policy",
  "selected_text": "図示は本文の構造を見える化する",

      "body": "hard gate の対象は visual-html-renderer のみでよいですか。reviewable-design-doc 側にも同じ禁止事項として入れるべきか確認したいです。",

comments.json のstatusは open、needs_agent_review、needs_user_reply、addressed、blocked、rejected、resolved に限定する。reply は role を user / agent / system、kind を note / clarification_request / implementation_note / status_change で表す。

レビューコメント取り込み

ユーザーが「レビュー終わったので確認して」「コメントを反映して」と言った場合、reviewable-design-doc は最新の preview session から対象HTML rootを特定し、annotations/comments.json を読み込む。反映できるコメントは設計へ取り込み、確認が必要なコメントは同じHTMLコメントスレッドへ agent reply として書き戻す。

自動取り込み対象

明確な修正依頼、誤字、章構成の指摘、図示追加の依頼、レビュー観点の不足は、そのまま設計モデルへ反映する。反映後はコメントstatusを addressed にし、agent replyで反映内容を短く残す。

確認が必要な対象

意図が複数に読める指摘、設計判断が分岐する指摘、既存制約と衝突する指摘は、チャットで聞き返すのではなく clarification_request の返信をコメントに追加し、statusを needs_user_reply にする。

処理フロー

  1. 最新の output/tmp/html-preview-sessions/*.json から成果物rootを特定する。
  2. annotations/comments.json を読み、未処理コメントとユーザー返信を抽出する。
  3. コメントを actionable, needs_clarification, blocked, already_addressed に分類する。
  4. actionable は設計モデルへ反映し、HTMLを再生成する。
  5. needs_clarification は該当コメントの replies に agent の確認質問を追加する。
  6. 更新済みHTMLとpreview URLを再提示し、確認待ちコメント数を報告する。
自動反映と確認返信は同じレビューサイクルで行う。agent がチャット本文に確認質問だけを書いて止まると、HTML上のレビュー文脈から外れるため、確認事項は必ずコメントスレッドにも残す。

実装・配布方式

この機能は SKILL.md だけで閉じるには処理が厚い。HTML生成、図レンダリング、preview server、コメント保存、レビュー取り込み、agent reply 書き戻し、検証を安定させる必要があるため、専用repoを作り Codex plugin として配布する構成を推奨する。

推奨: 専用repo + plugin

skill は判断フローと呼び出し規約だけを持ち、実処理は versioned script に寄せる。複数skillから同じ renderer / server / ingestion を呼べるため、既存HTML出力系skillへの横展開に向く。

推奨再利用テスト可能

非推奨: home/skills 直置きのみ

初期実装は速いが、scriptが増えると skill ごとに重複し、テスト・更新・配布の境界が曖昧になる。今回の規模では試作までに限定する。

試作向け長期運用は弱い

Data Contract

schemas/document-model.schema.json document-model.json。文書ID、タイトル、生成時刻、review settings、本文block配列を持つ。 render.py がHTML bundleを生成する入力契約にする。

script責務

script 責務 skillからの使い方
render.py 文書モデルからHTML bundleを生成する。テンプレート適用、asset配置、manifest作成を担当する。 visual-html-renderer が常に呼ぶ。
diagram_planner.py 文書モデルのブロックを図示候補へ分類し、Mermaid / SVG / draw.io / 画像生成の要求を正規化する。 図示が必要な場合だけ呼ぶ。

validate_bundle.py HTML存在、asset参照、図の非空、comment schema、preview bind方針を検証する。 両skillの完了前検証として呼ぶ。

plugin化の判断

観点 専用plugin repo home/skills直置き
script量 厚い処理をpackageとして管理できる。 skill配下にscriptが増え、見通しが悪くなる。
テスト fixtureつき単体テスト、bind検証、コメントschema検証を置ける。 手元検証に寄りやすい。
複数agent利用 Codex / Claude 向けのskillを同じ実装に載せやすい。 agent別に同期漏れが起きやすい。
導入速度 初期scaffoldは少し重い。 最短で試作できる。
推奨は、専用repoを作って plugin として visual-html-renderer と reviewable-design-doc を同梱する形。最初のMVPだけ既存 home/skills に置く場合でも、script API は plugin repo へ移せる境界で作る。

Preview Runtime

visual-html-renderer はHTML bundleを作った後、そのセッション中だけ有効なプレビューサーバーを起動し、ユーザーがクリックできるURLを提示する。URLを開くかどうかはユーザーが決めるため、ブラウザの自動起動はしない。

責務

成果物ディレクトリをHTTPで配信し、bind address、port、URL、PIDを session manifest に記録する。停止は明示的な終了処理またはセッション終了時に行う。

非責務

恒久daemon化、外部公開、認証つき共有、ブラウザ自動表示は扱わない。必要になった場合は別の公開・共有 skill として切り出す。

bind 方針

条件 bind address 提示URL 理由
tailscale ip -4 がIPv4を返す Tailscale IPv4 http://<tailscale-ip>:<port>/index.html VPN経由のSSH/tmux接続でも手元ブラウザから開ける。
Tailscale IPv4 が取れない 127.0.0.1 http://127.0.0.1:<port>/index.html ローカル端末内だけで完結する。
常時禁止 0.0.0.0 提示しない 意図しないLAN公開を避ける。

# 3. ユーザーにはURLだけ提示し、ブラウザは開かない
http://<bind-address>:<port>/index.html

# visual-html-renderer

## 役割
HTML出力系 skill の共通レンダラーとして、既存の個別HTML生成ロジックを置き換える。

## 実装方針
重い処理は plugin repo の scripts/html_review_workbench/ に置き、SKILL.md は入力確認、呼び出し順、ガード、検証手順に集中する。

## 基本手順
1. 文書モデルとレンダリングオプションを受け取る。
2. 図示・画像化した方が理解しやすいブロックを抽出する。
3. Mermaid / SVG / draw.io / 画像生成のどれを使うか決める。
4. HTML、CSS、図、必要な review UI 部品を生成する。
5. HTML構造、図の非空、responsive表示を検証する。
6. preview が有効なら空きportを選び、Tailscale IPv4が取れる場合はそのIP、なければ127.0.0.1でサーバーを起動する。
7. URL、PID、停止方法を提示し、session manifest に記録する。

## ガード
- レンダラーは内容判断を作らない。
- 図示は入力された構造を補助する目的に限定する。
- ブラウザを自動で開かない。URL提示までを責務とする。
- 0.0.0.0 にはbindしない。
- サーバーはセッション限定で起動し、恒久daemonにしない。
- 外部サービスへ投稿・アップロードする場合は別途承認ゲートを通す。

# reviewable-design-doc

## 役割
設計資料としてレビューできる構造を作り、最終HTML生成は visual-html-renderer に渡す。

## 実装方針
設計文書の構造化とレビュー取り込み判断を担当する。HTML生成、preview server、comment store の低レベル処理は plugin repo の共通scriptに委譲する。

## 基本手順
1. 設計対象、読者、レビュー目的、完了条件を整理する。
2. 要求、制約、アーキテクチャ、代替案、意思決定、未決事項へ分解する。
3. レビュー観点とコメントしてほしい範囲を明示する。
4. 図示要求と review mode を含む文書モデルを作る。
5. visual-html-renderer を使ってHTML bundleとpreview URLを生成する。
6. コメントJSONを次回修正の入力として扱える形で保存する。
7. レビュー完了後は comments.json を読み込み、明確な指摘は設計へ反映する。
8. 確認が必要な指摘は、該当コメントスレッドに agent reply を追加してHTMLを再生成する。

## ガード
- 設計として未確定の内容は確定事項と分けて書く。
- レビューコメント機能は必須で有効化する。
- 確認が必要な場合、チャットだけで聞き返さずHTMLコメントへ返信する。
- HTML低レベル実装をこのskillに重複実装しない。

Codex基盤契約

Milestone 0 では、pluginとして開発を継続できる最低限の契約をコードとテストで固定する。以後のMilestoneはこの契約を破らず、必要な拡張だけを追加する。

Codex manifest

.codex-plugin/plugin.json は name、version、description、skills、interface を持つ。interface には表示名、短い説明、長い説明、capabilities、defaultPrompt、brandColor を明示する。

CLI contract

scripts.html_review_workbench.cli は render、preview、ingest-review、validate を公開subcommandとして固定する。各skillはこの入口を呼び、低レベル実装へ直接依存しない。

CIで確認すること

  • python3 -m unittest discover -s tests でplugin構成、Codex manifest必須field、skill description、CLI契約を確認する。
  • python3 -m json.tool .codex-plugin/plugin.json >/dev/null と python3 -m json.tool .claude-plugin/plugin.json >/dev/null でmanifest JSON syntaxを確認する。
  • python3 -m scripts.html_review_workbench.cli --help >/dev/null でCLI entrypointが壊れていないことを確認する。

実装設計

部品 担当skill 実装方針 検証
Document Model reviewable-design-doc 設計資料を、requirements / constraints / architecture / decisions / open_questions / review_points に分解する。 章ごとに見出し、要旨、根拠が欠けていないか確認する。
Diagram Planner visual-html-renderer 図示候補を flow, architecture, matrix, timeline, concept に分類する。 図ごとに「本文のどの箇所を可視化したか」を保持する。
Renderer visual-html-renderer 自己完結HTMLを標準にし、必要時だけ外部asset分離にする。 ブラウザで開いてCSS/JS/図が欠落しないことを確認する。
Preview Runtime visual-html-renderer 成果物ディレクトリ用のセッション限定HTTPサーバーを起動する。Tailscale IPv4が取れる場合はそのIP、取れない場合は 127.0.0.1 にbindし、URLとPIDを manifest に残す。 提示URLで index.html が返ること、0.0.0.0 にbindしていないこと、ブラウザを自動起動していないことを確認する。
Comment Client visual-html-renderer Selection APIで選択範囲を取得し、block ID、選択テキスト、前後文脈、コメント、返信スレッド、statusを保存する。 コメント作成、agent reply表示、再読込、export JSON が動くことを確認する。
Review Server visual-html-renderer 任意でコメント保存APIを追加し、JSON保存だけを扱う。bind方針は Preview Runtime と共有する。 保存先が成果物ディレクトリ内であること、任意パス書き込みができないことを確認する。
Design Review Contract reviewable-design-doc 何をレビューしてほしいか、どの判断が未決か、どのコメントが修正対象かを明示する。 レビュー観点がHTML内に表示され、コメントJSONから修正対象へ戻れることを確認する。
Feedback Ingestion reviewable-design-doc レビュー完了発話を受けたら comments.json を読み込み、明確な指摘は設計へ反映し、確認が必要な指摘には同じコメントスレッドへ agent reply を追加する。 反映済みコメント、確認待ちコメント、ブロックコメントがstatusで区別され、更新HTMLに返信が表示されることを確認する。

検証計画

  1. Renderer contract 検証: visual-html-renderer が文書モデルからHTML bundleを生成できる。
  2. 図検証: 図がある場合、source と表示用SVG/PNGが両方存在し、空でない。
  3. Design doc 検証: reviewable-design-doc が要求、制約、意思決定、未決事項、レビュー観点を分離できる。
  4. コメント検証: 範囲選択からコメント登録、agent reply、一覧表示、JSON export が成功する。
  5. レビュー取り込み検証: 「レビュー終わったので確認して」発話から comments.json を読み、明確な指摘を反映し、確認が必要な指摘へHTMLコメント返信を書き戻せる。
  6. Preview Runtime 検証: Tailscale IPv4 がある場合はそのIP、ない場合は 127.0.0.1 のURLを提示し、0.0.0.0 にbindしていない。
  7. 境界検証: レンダラーが入力にない判断をHTMLや図で追加していない。ブラウザ自動起動や恒久daemon化も行っていない。

開発時のbacklog運用

実装はMilestone単位の docs/backlog/NN-*.md を起点に進める。各backlogは、作業中に docs/development-plan.html と本設計書 docs/design.html を同期することを完了条件に含める。

進捗の反映先

大きい進捗、Milestone順序、完了条件、リスクが変わった場合は docs/development-plan.html を更新する。

設計変更の反映先

skill範囲、script API、データ構造、Preview Runtime、コメント取り込み仕様が変わった場合は docs/design.html を更新する。

途中で設計が変わった場合、実装だけを進めない。開発計画と設計HTMLの両方を直し、backlogの進捗ログに更新箇所を残してから作業を続ける。

段階的な作り方

段階 作るもの 完了条件
Phase 0 専用repo / plugin scaffold。.codex-plugin/plugin.json、2つのskill、共通script package、fixture testを置く。 pluginとして読み込める最小構成があり、script CLIの入出力契約が固定される。
Phase 1 visual-html-renderer の最小版。自己完結HTMLテンプレート、図示枠、responsive CSS。 既存HTML出力の代替として、同じ入力から図示つきHTMLを生成できる。
Phase 2 visual-html-renderer の図示強化。Mermaid source保存、SVGレンダリング、draw.io optional export。 図のsourceと表示結果が成果物内に残り、HTMLから参照できる。
Phase 3 visual-html-renderer の Preview Runtime。Tailscale IPv4優先bind、fallback localhost、URL提示、session manifest。 SSH/tmux環境でも提示URLをクリックして開ける。ブラウザ自動起動や 0.0.0.0 bind がない。
Phase 4 reviewable-design-doc の最小版。設計資料構造、レビュー観点、standaloneコメント。 レビュー可能な設計資料HTMLを生成し、preview URLまで提示できる。
Phase 5 review-server mode と comments.json 永続保存。 コメントをagentが次作業の入力として読める。
Phase 6 レビュー取り込みと agent reply。レビュー完了発話、コメント分類、設計反映、確認質問のコメント返信。 確認が必要な指摘に対し、チャットだけでなくHTMLコメント上にagent返信が残る。
Phase 7 既存の調査・設計系skillから visual-html-renderer を呼び出すよう移行。 HTML出力系 skill が個別テンプレートではなく共通レンダラーを使う。
